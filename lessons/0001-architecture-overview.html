The Two Halves

The Three Databases

Skribble uses a dual-database pattern (decided in ADR-0001):

Store Purpose What's Inside
MongoDB Persistent data User accounts, room metadata (for the public feed)
Redis Ephemeral game state Active rooms, player lists, drawing data, scores
localStorage Client persistence JWT token, user profile (survives page reloads)
Why two databases? MongoDB is durable but slower. Redis is fast but ephemeral. Game state needs to be fast (players are actively drawing and guessing), so Redis handles it. User accounts need to survive server restarts, so MongoDB handles them.

How Data Flows

There are exactly two communication channels between client and server:

1. REST API (HTTP) — For one-shot operations: register, login, create room, fetch feed, update profile, update stats. Request-response. Used before and after games.
2. Socket.IO (WebSocket) — For real-time game operations: join room, draw, guess, word selection, hints, vote kick. Event-driven. Used during games.

The Game State Machine

Every room follows this lifecycle:

NOT_STARTED ──► CHOOSING_WORD ──► DRAWING ──┐ ▲ │ │ ┌──────────────────────────┘ │ │ │ ├──► (all guessed or time up) │ │ │ ▼ │ TURN_END ──► (next turn?) ──► CHOOSING_WORD │ (last turn?) │ │ │ ▼ └──────────────── GAME_ENDED

Each state transition is triggered by a socket event or timer, and is handled in roomController.ts.
