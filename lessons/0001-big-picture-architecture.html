What is Skribble?

Skribble is a real-time multiplayer drawing-and-guessing game (like skribbl.io). Players join rooms, take turns drawing a secret word, and race to guess it first. The project is split into two independent processes that communicate over WebSocket (via Socket.IO):

How the Processes Talk

There is no REST API for game actions. Everything flows through Socket.IO events:

  1. Client emits an event (e.g., "guess" with payload { guess: "elephant" })
  2. Server handler receives it, processes it (checks word match, scores, updates Redis)
  3. Server emits back to the room (e.g., "guessed" with the player who guessed correctly)
  4. Client listeners in RoomContext/MessagesContext update React state, which re-renders components

Key insight: The server is the source of truth. Redis persists room state. Clients are stateless projections — they only know what the server tells them via events.

The Room Object — Central Data Structure

The Room interface is the single most important type. It's defined identically on both server and client:

interface Room {
  roomId: string;          // UUID v4
  creator: string | null;  // Socket ID of room creator
  players: Player[];       // Array of players in room
  gameState: GameState;    // Current round, word, drawing data, hints
  settings: Settings;      // Configurable game parameters
  isPrivate: boolean;      // Private room or public matchmaking
  vote_kickers: [string, string[]][]; // Vote-kick tracking
}

When the server modifies a room (e.g., a player guesses), it writes the entire Room object to Redis, then emits the relevant event to the Socket.IO room so clients can update.

Public vs Private Rooms

  • Private rooms: Created with isPrivate=true. The creator is the host. Settings can be changed. Game starts manually.
  • Public rooms: Matchmaking. When a player clicks "Play!", the server looks for an existing public room with space. If none found, creates a new one. Game starts automatically when 2+ players are present.

The key distinction is in Redis: private rooms use key room:<id>, public rooms use publicRoom:<id>. The getPublicRoom() function scans for available public rooms by language.

First Principles: Socket.IO Rooms vs Game Rooms

Don't confuse the two meanings of "room":

  • Socket.IO room: A channel that the Socket.IO server manages. When a socket join("abc123"), it will receive all events emitted to io.to("abc123").
  • Game room: A logical Room object stored in Redis. It has settings, players, and game state.

Both use the same roomId string. When a player joins a game room, their socket also joins the Socket.IO room with that ID — this is how broadcast works.

💡 Interview Question: "Why use Redis instead of in-memory state?"

Good answer: Redis makes the server horizontally scalable — multiple server instances can share room state. It also persists room data across server restarts. The trade-off is added infrastructure complexity (Docker container).
