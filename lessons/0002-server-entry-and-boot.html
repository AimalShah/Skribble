Server Entry & Boot

How the server starts up, connects to databases, and wires everything together.

What You'll Learn

By the end of this lesson, you'll know exactly what happens when you run npm run dev in the server — every function call, every connection, every setup step.

The Entry Point

server/src/server.ts is where it all begins. Here's the boot sequence:

// 1. Create Express app + HTTP server
const app = express();
const server = http.createServer(app);

// 2. Configure middleware
app.use(cors());
app.use(express.json());

// 3. Mount REST API routes
app.use("/api/auth", authRoutes);
app.use("/api/profile", profileRoutes);
app.use("/api/rooms", roomRoutes);
app.use("/api/stats", statsRoutes);

// 4. Health check endpoint
app.get("/api/health", (req, res) => res.json({ status: "ok" }));

// 5. Connect to MongoDB
await mongoose.connect(MONGO_URI);

// 6. Create Socket.IO server
const io = new Server(server, { cors: { origin: "*" } });

// 7. Register socket event handlers
setupSocket(io);

// 8. Start CLI command listener
setupCommandLine(io);

// 9. Listen on port 8000
server.listen(8000);
Key function: createApp() The file exports createApp() which returns { app, server } without starting the listener. This is for testing — tests can use the Express app with supertest without binding to a port.

Core Interfaces

interface Player {
  playerId: string;
  name: string;
  appearance: [number, number, number]; // [body, eyes, mouth]
  score: number;
  guessed: boolean;
  guessedAt?: number;
}

interface GameState {
  currentRound: number;
  drawingData: DrawData[];      // every stroke point
  guessedWords: string[];
  word: string;                  // the secret word
  currentPlayer: number;         // index into players[]
  hintLetters: GuessedLetters[];
  roomState: RoomState;
  timerStartedAt?: number;
}

interface Room {
  roomId: string;
  creator: string;               // playerId of room owner
  players: Player[];
  gameState: GameState;
  settings: Settings;
  isPrivate: boolean;
  vote_kickers: Record<string, string[]>; // target -> voter IDs
}
Watch out: RounEndReason is a typo in the codebase (missing 'd' in "Round"). It's RounEndReason, not RoundEndReason. You'll see this consistently throughout the code.

The Models

User Model (server/src/models/User.ts)

Stores persistent user data in MongoDB:

{
  email: string (unique),
  password: string (bcrypt hashed, select: false),
  displayName: string,
  avatar: { body: number, eyes: number, mouth: number },
  stats: {
    gamesPlayed: number,
    gamesWon: number,
    totalScore: number,
    drawingRounds: number,
    wordsGuessed: number
  },
  createdAt: Date
}

The pre('save') hook automatically hashes passwords with bcrypt (10 salt rounds) before saving. The comparePassword() method verifies passwords during login.

RoomMetadata Model (server/src/models/RoomMetadata.ts)

Stores room display data for the public feed:

{
  roomId: string (unique, indexed),
  ownerId: ObjectId (ref User),
  name: string,
  isPrivate: boolean,
  passwordHash?: string,
  published: boolean,
  language: string,
  settings: Settings,
  playerCount: number,
  gameState: "NOT_STARTED" | "IN_PROGRESS" | "ENDED",
  createdAt: Date
}
Why two room stores? RoomMetadata in MongoDB is for the public feed — it persists so users can browse rooms. The actual game state (players, scores, drawing data) lives in Redis and is ephemeral.
