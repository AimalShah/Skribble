How RoomContext Works

The RoomContext.tsx is the heart of client state. Let's trace its key parts:

State Declaration

const [room, setRoom] = useState<Room>(/* empty initial state */);
const [myTurn, setIsmyTrun] = useState(false);
const [me, setMe] = useState<Player | null>(null);
const [roomState, setRoomState] = useState<RoomState>(RoomState.NOT_STARTED);
const [mutedPlayers, setMutedPlayers] = useState<string[]>([]);

Socket Event Subscription

useEffect(() => {
  socket.on(GameEvent.JOINED_ROOM, joinedRoom);
  socket.on(GameEvent.WORD_CHOSEN, wordChosen);
  socket.on(GameEvent.TURN_END, setTurn);
  socket.on(GameEvent.GAME_STARTED, gameStarted);
  // ... 7 more listeners
  return () => { /* cleanup: socket.off(...) */ };
}, []);

Key Handler: joinedRoom()

function joinedRoom(room: Room) {
  setRoom(room);                                              // Store the full room object
  setMe(room.players.find(p => p.playerId === socket.id) ?? null); // Identify "me"
}

Key Handler: gameStarted()

function gameStarted(room: Room) {
  const cP = room.players[room.gameState.currentPlayer] || null;
  if (cP && socket.id === cP.playerId) setIsmyTrun(true);   // Am I the drawer?
  else setIsmyTrun(false);
  joinedRoom(room);                                          // Store room + find "me"
}

Derived State

const currentPlayer = room.players[room.gameState.currentPlayer] || null;

This is computed on every render — currentPlayer is derived from the room data, not stored separately.

Key insight: The myTurn flag is what controls whether the Toolbar (colors, brush sizes, undo/clear) is rendered. It's also used by GameCanvas to gate drawing events. Check the Toolbar.tsx render: if (!myTurn || roomState !== RoomState.DRAWING) return null;

How MessagesContext Works

MessagesContext is simpler — it just maintains an array of IMessage objects. Each message has a type enum that determines how it's rendered:

enum MessageType {
  Guess = "guess",          // "PlayerName: hello"
  PlayerLeft = "playerLeft", // "PlayerName left the game"
  PlayerJoin = "playerJoin", // "PlayerName joined the game"
  WordGuessed = "wordGuessed", // "PlayerName has guessed the word"
  GuessClose = "guessClose", // "'hello' is close" (unused—no close logic on server)
  WordChoosen = "wordChosen", // "PlayerName is now drawing"
  WordWas = "wordWas",       // "The word was 'elephant'"
  Error = "error",           // Error messages
  VoteKick = "voteKick",     // "X is voting to kick Y (2/4)"
}

The Message component uses these types to style each message differently — green for success, red for errors/warnings.

Important: There's a subtle issue in the cleanup — both useEffect blocks in MessagesContext.tsx use socket.on with no cleanup for some events. In React StrictMode, this causes duplicate listeners.

Key Components — Their Jobs

The socketHandler.ts Singleton

import io from "socket.io-client";
export const socket = io("http://192.168.29.63:8000", { autoConnect: false });

autoConnect: false means we manually call socket.connect() when the user clicks Play. This prevents connecting before the user is ready. The IP is hardcoded — this is a development setup.

💡 Interview Question: "Why use React Context instead of Redux or Zustand?"

The app has a simple state shape (one Room object + one messages array). Context is built into React, has zero dependencies, and is sufficient for this scale. Redux would be over-engineering. The downside is that any context update re-renders all consumers, but with the component tree being small, this isn't a problem.
