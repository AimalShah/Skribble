Architecture: Drawing Data Flow

The drawing system is straightforward. Let's trace a single stroke:

Drawer's Canvas (mousemove) │ │ socket.emit("draw", { x, y, color, lineWidth, end }) ▼ Server (socketHandlers.ts) │ │ handleDrawAction("DRAW") checks: │ - Is game running? (currentRound > 0) │ - Is this the current drawer? (playerId match) │ │ socket.to(roomId).emit("drawData", drawData) │ room.gameState.drawingData.push(drawData) │ setRedisRoom(...) ▼ All Other Players' Canvases │ │ socket.on("drawData", revieveDrawData) // sic │ Draws the stroke on their local canvas │ ▼ Late-joiners: drawingData[] is replayed via loadDrawData() on JOINED_ROOM + GAME_STATE events

GameCanvas Component — Deep Dive

The canvas is a plain HTML5 canvas (no library like Fabric.js or react-canvas-draw, despite those being in package.json):

const GameCanvas = ({ room }: { room: Room }) => {
  const canvasRef = useRef<HTMLCanvasElement>(null);
  const [lineWidth, setLineWidth] = useState<number>(5);
  const { myTurn } = useRoom();
  const [color, setColor] = useState<string>("#000000");
  let drawing = false;                 // <-- NOT state! Just a local variable
  const drawData = useRef<DrawData[]>([]);

  // ... event handlers
};

⚠️ Bug alert: drawing is a plain variable, not state. This means it resets to false on every re-render. However, since React event handlers use closures over the last render, this actually works because startDrawing sets it to true via the event handler closure. The real risk is that stale closures could lose the drawing = true assignment if a re-render occurs mid-draw — but since canvas events don't trigger re-renders, it works in practice.

Getting Coordinates

function getCoords(event: MouseEvent | TouchEvent) {
  const rect = canvasRef.current.getBoundingClientRect();
  const point = "touches" in event ? event.touches[0] : event;
  return {
    x: ((point.clientX - rect.left) * canvasRef.current.width) / rect.width,
    y: ((point.clientY - rect.top) * canvasRef.current.height) / rect.height,
  };
}

This scales coordinates to the canvas's internal resolution (800×600). Without this scaling, touch/mouse coordinates would be in viewport pixels, which differ when the canvas is CSS-scaled to fill its container.

Drawing (Drawer's Perspective)

function draw(event: MouseEvent | TouchEvent) {
  if (!drawing || !canvasRef.current) return;
  const { x, y } = getCoords(event);
  const ctx = canvasRef.current.getContext("2d");

  ctx.lineWidth = lineWidth;
  ctx.lineCap = "round";
  ctx.strokeStyle = color;
  ctx.lineTo(x, y);
  ctx.stroke();
  ctx.beginPath();
  ctx.moveTo(x, y);

  socket.emit(GameEvent.DRAW, { x, y, color, lineWidth });
  drawData.current.push({ x, y, color, lineWidth, end: false });
}

Each mousemove event emits a DRAW event with the current point. The canvas draws a line from the previous point to the current one. lineCap = "round" makes the stroke smooth.

Ending a Stroke

function stopDrawing() {
  if (!drawing) return;
  drawing = false;
  const ctx = canvasRef.current?.getContext("2d");
  if (ctx) ctx.beginPath();            // Stop the path — next stroke starts fresh

  // Mark the last segment as end=true so receivers know to beginPath
  if (drawData.current.length > 0) {
    const lastDrawData = drawData.current[drawData.current.length - 1];
    lastDrawData.end = true;
    socket.emit(GameEvent.DRAW, lastDrawData);  // Re-send with end flag
  }
}

The end flag is crucial. On the receiver's side, when end=true, they call ctx.beginPath() to lift the pen — otherwise strokes would connect across mouse-up events.

Receiving Draw Data (Other Players)

function revieveDrawData(data: DrawData) {
  const ctx = canvasRef.current.getContext("2d");
  ctx.lineWidth = data.lineWidth;
  ctx.lineCap = "round";
  ctx.strokeStyle = data.color;
  ctx.lineTo(data.x, data.y);
  ctx.stroke();
  ctx.beginPath();
  ctx.moveTo(data.x, data.y);
  if (data.end) ctx.beginPath();  // Lift pen on end of stroke
}

Undo Operation

The undo is handled on both sides:

// Client-side (GameCanvas.tsx)
function handleUndo() {
  // Pop from drawData ref until we hit the stroke boundary (end=true)
  drawData.current.pop();
  while (drawData.current.length > 0) {
    const data = drawData.current.pop();
    if (data?.end) break;
  }
  // Re-mark the new last point as end=true
  if (drawData.current.length > 0)
    drawData.current[drawData.current.length - 1].end = true;

  clearCanvas();
  loadDrawData();  // Redraw everything from scratch minus the undone stroke
}

// Server-side (roomController.ts)
export async function handleDrawAction(socket, action, drawData?) {
  // ...
  case "UNDO":
    room.gameState.drawingData.pop();  // Remove last draw data
    socket.to(room.roomId).emit(GameEvent.UNDO_DRAW);
    break;
}

Important: The server only pops ONE element from drawingData, but the client pops the entire last stroke (multiple segments). This means after undo, the server's drawingData array is out of sync with the client if a late-joiner replays it. The undo is purely optimistic on the client side.

Clear Operation

// Client
function clearCanvas() {
  const ctx = canvasRef.current.getContext("2d");
  ctx.clearRect(0, 0, canvasRef.current.width, canvasRef.current.height);
}

// Server
case "CLEAR":
  room.gameState.drawingData = [];  // Wipe all stored data
  socket.to(room.roomId).emit(GameEvent.CLEAR_DRAW);
  break;

Loading Saved Drawings

When a player joins mid-game (via handleInBetweenJoin), the server sends the GAME_STATE event with the current gameState. The GameCanvas listens for GAME_STATE and replays the stored drawingData:

useEffect(() => {
  socket.on(GameEvent.JOINED_ROOM, loadDrawData);     // Load on join
  socket.on(GameEvent.DRAW_DATA, revieveDrawData);     // Receive live
  socket.on(GameEvent.WORD_CHOSEN, clearCanvas);       // Clear on new round
  socket.on(GameEvent.CLEAR_DRAW, clearCanvas);        // Clear on explicit clear
  socket.on(GameEvent.UNDO_DRAW, clearCanvas);          // <-- BUG: reloads all data instead of just clearing
}, []);

⚠️ Bug alert: Listening to UNDO_DRAW and calling clearCanvas means when the drawer undoes, all other players see a blank canvas but the drawing data is NOT re-loaded. The server doesn't re-emit the full drawingData after undo. This is a known bug — the undo only works correctly on the drawer's own canvas.

Toolbar

The Toolbar component (Toolbar.tsx) is only rendered when myTurn === true && roomState === RoomState.DRAWING. It provides:

  • Color palette: 26 colors in a 2×13 grid. Primary/secondary color swatch with swap button.
  • Line width selector: 6 sizes (5, 7, 12, 14, 16, 20). Popup menu with visual circles.
  • Undo button: Emits DRAW_UNDO + calls local handleUndo()
  • Clear button: Emits DRAW_CLEAR + calls local clearCanvas()

Under the hood: The colors array has two rows — light and dark. The first color is white (eraser effect). The primary/secondary swatch uses a CSS gradient split diagonally.

💡 Interview Question: "How does the drawing sync work in real-time?"

Each mousemove event generates a { x, y, color, lineWidth, end } segment that's emitted to the server. The server validates the sender is the current drawer, stores the segment, and relays it to all other players in the room. The receiving clients render each segment on their canvas as it arrives. Late-joiners replay the entire drawingData[] array to catch up.
