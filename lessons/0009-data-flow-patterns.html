Data Flow Patterns

Trace complete user actions from click to state change — the full journey through the system.

What You'll Learn

By the end of this lesson, you'll be able to trace any user action through every layer of the system — from button click to database write and back.

Flow 1: User Registers

Client Server ────── ────── AuthForm submit → AuthContext.register() → POST /api/auth/register ─────────────────────────────► auth.ts: register() → Validate fields → Check email exists → 409 → User.create() [bcrypt hook] → signToken(userId) ◄──────── {token, user} ────── → localStorage.setItem(token) → AuthContext updates state → App re-renders, ProtectedRoute passes

Flow 2: User Creates Room

Client Server MongoDB Redis ────── ────── ─────── ───── CreateRoomPage submit → POST /api/rooms ─────────────────────────────► rooms.ts: create() → generateRoomId() → bcrypt hash password → RoomMetadata.create() ────────────────────► SAVE ← {roomId, metadata} ◄──────────────────────────── [Optional] POST /rooms/:id/publish ─────────────────────────────► rooms.ts: publish() → set published=true ← {success} → Navigate to /room/:roomId → GameRoomPage mounts → socket.connect() → emit JOIN_ROOM ─────────────────────────────► socketHandlers.ts → handleNewRoom() → generateEmptyRoom() → setRedisRoom() ─────────────────────────► SAVE ◄──────── JOINED_ROOM(room) ── → RoomContext sets room → Game renders

Flow 3: Player Draws a Stroke

Drawer's Client Server Other Clients ────────────── ────── ────────────── Canvas mousemove → draw() called → emit DRAW {x,y,color,w} ──────────────────────────► roomController.ts → withRoomLock() → Validate sender=drawer → Push to drawingData[] → setRedisRoom() → broadcast DRAW_DATA ─────────────────────────────► revieveDrawData() → ctx.lineTo(x,y) → ctx.stroke()

Flow 4: Player Guesses Correctly

Guesser's Client Server All Clients ──────────────── ────── ─────────── Chat submit "apple" → emit GUESS {guess:"apple"} ──────────────────────────► roomController.ts → withRoomLock() → guess === word? → YES: → player.guessed = true → player.guessedAt = now → emit GUESSED {sender, guess} ─────────────────────────────► → Check: all guessed? → NO: continue → YES: → givePoints() → endRound() → emit TURN_END {room, word} ─────────────────────────────► Canvas clears Overlay shows word (5s delay) → nextRound()

Flow 5: Game Ends

Server All Clients ────── ─────────── endGame(roomId, io) → Reset roomState=NOT_STARTED → Reset all scores → emit GAME_ENDED {room} ──────────────────────────► RoomContext: WINNER state OverlayContent: GameRoomPage: → POST /api/stats/update → {gamesPlayed++, etc.} (10s delay) → For public rooms: → startGameTimers set → Auto-restart game

The Dual-Database Pattern

┌─────────────────────────────────────┐ │ MongoDB │ │ (persistent, slower, durable) │ │ │ │ Users collection: │ │ → email, password, avatar, stats │ │ │ │ RoomMetadata collection: │ │ → roomId, name, published, │ │ settings, language │ │ │ │ Used for: REST API only │ │ (auth, profile, feed, stats) │ └──────────────┬───────────────────────┘ │ REST API calls (before/after games) │ ┌──────────────┴───────────────────────┐ │ Redis │ │ (ephemeral, fast, volatile) │ │ │ │ "room:{id}" or "publicRoom:{id}": │ │ → players[], gameState, │ │ drawingData[], settings │ │ │ │ Used for: Socket.IO only │ │ (all game operations) │ └─────────────────────────────────────┘
